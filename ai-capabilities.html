AI 目前还搞不定的三件事

了解边界，比了解能力更重要——这些「坑」你需要提前知道。

⚠️

幻觉问题 — 精确数据不可靠

AI 在需要精确数据统计数据、财务数字、法律条文、历史日期等的任务中可信度不高。它会「自信地」编造事实、虚构引用、捏造数据——而这一切看起来都像是真的。

根本原因：大模型本质是「概率预测下一个 token」，不是「检索事实」。它更擅长生成看起来合理的内容，而非事实上正确的内容。

应对策略：涉及精确数据时，主动要求 AI 标注信息来源；关键事实必须二次核实；搭配 RAG（检索增强生成）可显著降低幻觉率。
📏

上下文限制 — 长链条任务容错率低

当前模型的上下文窗口虽然在不断扩大，但在压缩上下文时仍容易丢失关键信息。对于需要多步推理、跨文件协作的复杂长链条任务，中间环节的微小偏差会在后续步骤中被放大。

典型表现：处理到后半段时「忘记」了前面的约束条件；多轮对话中逐渐偏离初始目标；长文档处理时遗漏关键段落。

应对策略：将大任务拆解为独立小任务，每步输出后人工确认；关键约束在每轮对话中重申；善用外部记忆（文件、数据库）作为状态锚点。
🐛

编程「快但不好」 — 隐性质量问题突出

AI 写代码确实快，但质量参差不齐。它经常忽略边界条件、异常处理和安全漏洞，甚至会虚构 API 参数编造不存在的函数签名、参数名或返回值、调用不存在的接口。

三个高频问题：① 空值/边界不处理 ② SQL 注入 / XSS 漏洞 ③ 使用已废弃或幻觉生成的 API。

应对策略：AI 生成的代码必须经过 Code Review；关键逻辑编写单元测试；使用静态分析工具（ESLint / SonarQube）做第一道防线；安全敏感代码必须人工把关。
